# Admin UI System Plan

## Мета

Повернути адмінку `MIL SPIL Feed Generator` до єдиної дизайн-системи:

- без емодзі й випадкових стилів;
- без змішування різних патернів у межах однієї вкладки;
- з окремими вкладками під канали;
- з повторно вживаними UI-компонентами;
- зі зрозумілою структурою для майбутніх інтеграцій.

## Поточна проблема

База адмінки добра:

- є власний header;
- є токени в `assets/admin.css`;
- є таби на SVG-іконках;
- є власні card-компоненти.

Але нові секції почали додаватись нерівномірно:

- частина розмітки тримається на системі токенів;
- частина вже на inline-style;
- у кнопках і статусах з'явилися емодзі;
- одна вкладка містить кілька окремих сценаріїв;
- `Prom.ua Sync` змішує Prom import, Prom export і supplier feeds;
- `Фільтри` і `Маппінг` винесені в top-level, хоча вони не канали.

## Цільова інформаційна архітектура

### Top-level вкладки

1. `Огляд`
2. `Google`
3. `Prom`
4. `Постачальники`
5. `Rozetka`
6. `Facebook`
7. `Налаштування`
8. `Довідка`

## Логіка вкладок

### `Огляд`

Тут мають бути:

- короткий статус усіх каналів;
- останні запуски;
- помилки;
- системний стан;
- швидкі дії `Запустити зараз`.

### `Google`

Тут мають бути:

- GMC feed;
- мовні фіди;
- тестова валідація;
- Google Product Category;
- Custom Labels;
- Google-специфічні фільтри.

### `Prom`

Тут мають бути:

- API token;
- імпорт `Prom -> WooCommerce`;
- експорт `WooCommerce -> Prom`;
- schedule;
- realtime sync;
- останні імпорт/експорт summary.

### `Постачальники`

Тут мають бути:

- supplier feeds;
- список джерел;
- schedule;
- ручний запуск;
- результати останніх імпортів;
- надалі per-feed schedules.

### `Rozetka`

Окрема вкладка навіть до повної реалізації. Вона потрібна як stable entry point для:

- налаштувань каналу;
- специфіки формату;
- статусу генерації;
- майбутнього експорту або sync.

### `Facebook`

Окрема вкладка для:

- Meta / Facebook catalog feed;
- налаштувань полів;
- schedule;
- відбору товарів.

### `Налаштування`

Лише глобальні речі, спільні для всіх каналів:

- бренд;
- країна походження;
- часовий пояс;
- WebP -> JPG;
- глобальна генерація;
- технічні дефолти.

### `Довідка`

Лише документація, підказки, workflow і пояснення.

## Що треба винести з top-level

### `Маппінг`

Не повинен бути окремою top-level вкладкою після рефакторингу.

Його треба перенести в:

- `Google`, якщо маппінг стосується feed для GMC;
- або `Налаштування`, якщо з'явиться спільний mapping engine для кількох каналів.

Поточний стан:

- `Google Product Category` і `Custom Labels` сильно прив'язані саме до Google.

Тому базовий варіант:

- перенести `Маппінг` у вкладку `Google`.

### `Фільтри`

Не повинні бути top-level вкладкою, якщо вони не окремий продукт.

Кращий варіант:

- глобальні фільтри лишити в `Налаштування`;
- канал-специфічні фільтри тримати у відповідних вкладках каналів.

## Правило для майбутніх каналів

Окрема вкладка потрібна, якщо є хоча б одна з умов:

- канал має власний API або auth;
- канал має власний import/export flow;
- канал має власний schedule;
- канал має більше 5 окремих полів налаштувань;
- канал має власну статистику запусків;
- канал має власні правила маппінгу або фільтрації.

Якщо цього немає, новий функціонал не повинен створювати нову вкладку.

## Структура всередині вкладки

Кожна вкладка має складатися з блоків у такому порядку:

1. `Intro card`
2. `Settings cards`
3. `Action card`
4. `Summary / status card`
5. `Danger / warning card`, якщо потрібно

Не можна змішувати в одній card:

- довгу форму налаштувань;
- оперативні кнопки запуску;
- результати останнього запуску;
- довідковий текст на пів сторінки.

Адмінка повинна спиратися на сталий набір компонентів:

- `d14k-subsection-title`
- `d14k-action-bar`

Якщо новий екран не вкладається в ці компоненти, спочатку треба додати новий загальний компонент у CSS, а вже потім використовувати його в PHP.

## Перший етап рефакторингу

### Етап 1

- прибрати емодзі з `class-admin-settings.php`;
- перевести status/summary/buttons на SVG + CSS state classes;
- прибрати inline-style з нових секцій;
- добити єдиний pattern для action/result/summary.

### Етап 2

- розбити `Prom.ua Sync` на `Prom` і `Постачальники`;
- створити нову вкладку `Огляд`;
- перемістити `Маппінг` у `Google`;
- перемістити `Фільтри` в `Google` або `Налаштування`.

### Етап 3

- підготувати пусті вкладки `Rozetka` і `Facebook`;
- дати їм consistent empty state замість довільного `coming soon`.

## Другий етап рефакторингу

- винести повторювані шматки PHP-розмітки в helper methods;
- зменшити обсяг `render_page()`;
- прибрати випадкові ручні `margin-top`, `gap`, `style=""` із шаблону;
- сформувати channel-specific renderer methods:
  - `render_overview_tab()`
  - `render_google_tab()`
  - `render_prom_tab()`
  - `render_supplier_tab()`
  - `render_rozetka_tab()`
  - `render_facebook_tab()`
  - `render_settings_tab()`
  - `render_help_tab()`

## Definition of Done для UI-змін

UI-зміна вважається завершеною, якщо:

- не додано жодного емодзі в UI;
- не додано нових inline-style без крайньої причини;
- нова секція використовує існуючі design tokens;
- нова секція має однаковий ритм відступів із сусідніми;
- нова секція має зрозумілий header, body і actions;
- нова секція не дублює патерн, який уже існує під іншою назвою;
- новий функціонал потрапив у правильну вкладку за логікою каналу;
- вигляд перевірено в desktop і вузькому viewport.
