# Admin UI Style Rules

## Для чого цей файл

Перед будь-яким розширенням адмінки плагіна треба перечитати цей файл.

Мета:

- не повертатися до хаотичних правок;
- тримати єдиний стиль;
- не засмічувати `class-admin-settings.php`;
- не плодити нові патерни без потреби.

## Головні правила

1. Не використовувати емодзі в UI.
2. Не додавати inline-style в PHP, якщо це не тимчасова відладка.
3. Не створювати новий патерн кнопки, badge або notice, якщо вже є близький.
4. Не змішувати кілька сценаріїв у межах однієї card.
5. Не додавати нову вкладку без перевірки правил з `admin-ui-system-plan.md`.

## Іконки

### Треба

- використовувати SVG;
- тримати всі іконки в одному stroke-style;
- ставити однаковий розмір у межах одного рівня UI.

### Не треба

- емодзі типу `✅`, `⚠`, `🔄`, `⬇️`, `⬆️`, `💾`;
- dashicons як випадковий fallback усередині нових секцій;
- мікс різних стилів SVG в одному блоці.

## Кольори

Усі кольори мають братися тільки з токенів у `:root` в `assets/admin.css`.

### Не можна

- писати сирі кольори в PHP;
- заводити новий колір під одну конкретну кнопку;
- робити локальні `style="color:#..."`.

### Якщо потрібен новий колір

1. Додати його в design tokens.
2. Дати йому ясне системне ім'я.
3. Використовувати через CSS class, а не напряму.

## Відступи

Відступи мають бути системними.

### Базовий ритм

- між вкладками і першим блоком: великий відступ;
- між cards: середній відступ;
- між header card і body: малий відступ;
- між полями всередині секції: сталий відступ;
- між summary line items: малий відступ.

### Не можна

- ставити випадкові `margin-top: 10px`, `margin-bottom: 16px`, `gap: 12px` прямо в PHP;
- вирівнювати блоки через ручні `br` або порожні `p`;
- змішувати різні вертикальні ритми в сусідніх секціях.

## Форми

### Правило

Усі field rows мають користуватися або:

- стандартним `form-table`, якщо це звичайні налаштування;
- або власним компонентом групи полів.

### Не можна

- робити кожен новий field row вручну з окремими inline-width;
- вставляти кнопки в полі без єдиного wrapper;
- вигадувати окремий layout для кожного каналу.

### Треба

- для горизонтальних груп використовувати `d14k-inline-actions`;
- для вертикальних груп використовувати `d14k-field-stack`.

## Action blocks

Кнопки запуску й результати мають жити в `d14k-action-bar`.

### Має бути

- primary action;
- secondary action за потреби;
- `d14k-action-result` для статусу;
- однакова поведінка pending / success / error.

### Не можна

- фарбувати result через jQuery inline-style;
- робити окремий layout result для кожної секції;
- ліпити кнопку `Save` окремою пустою card.

## Status і summary

### Статуси

Статуси мають показуватись через:

- `d14k-badge`
- `d14k-status-badge`
- `d14k-notice-inline`
- `d14k-notice-box`

### Summary

Останній запуск, результат імпорту, результат експорту і статистика мають жити в:

- `d14k-sync-summary`
- `d14k-sync-summary-metrics`
- `d14k-sync-summary-list`

### Не можна

- писати голий текст зі статусом прямо під кнопкою;
- використовувати емодзі замість системного state style;
- змішувати success text і error text без візуального контейнера.

## Таблиці

### Коли можна table

- список фідів;
- системний статус;
- простий data grid;
- mapping rows, якщо це реально табличні дані.

### Коли не можна table

- для layout сторінки;
- для action blocks;
- для notice sections;
- для mixed content cards.

## Тексти

### Заголовки

- короткі;
- один змістовий центр на card;
- без емодзі.

### Пояснення

- 1 короткий абзац на початку card;
- без дублювання того, що видно з полів;
- без перевантаження термінами.

### Description text

- короткий;
- конкретний;
- без розтягнутих технічних пояснень у кілька абзаців.

## CSS naming

### Використовувати

- `d14k-block`
- `d14k-block__element`, якщо буде потреба
- `d14k-block--modifier`
- `is-success`, `is-error`, `is-pending` для станів

### Не використовувати

- одноразові назви під конкретний екран;
- імена типу `new-box`, `temp-row`, `second-style`;
- стилі, прив'язані до тексту заголовка.

## Заборонено без окремого обґрунтування

- `style=""` у PHP
- емодзі в UI
- нові сирі кольори
- нові локальні тіні
- нові локальні border-radius
- кнопки з випадковим icon placement
- нові компоненти без CSS class

## Дозволено, якщо справді треба

- inline-style лише для тимчасової діагностики під час розробки, але не в фінальному коді;
- локальний style в demo/placeholder, який не йде в реліз;
- окремий layout-компонент, якщо старі не покривають сценарій і це оформлено як загальний reusable class.

## Чеклист перед кожною UI-правкою

1. Яка це вкладка за логікою каналу?
2. Чи є вже схожий компонент у CSS?
3. Чи можна зібрати секцію з `d14k-card`, `d14k-action-bar`, `d14k-sync-summary`?
4. Чи немає тут емодзі?
5. Чи немає тут inline-style?
6. Чи витримано однаковий ритм відступів?
7. Чи не змішані налаштування, дії і статус в одному контейнері?
8. Чи зрозуміло, що тут primary action?
9. Чи виглядатиме це нормально на вузькому екрані?
10. Чи не треба спочатку оновити `admin-ui-system-plan.md`?

## Правило для майбутніх pull-style змін

Кожна нова UI-фіча повинна містити:

- коротке пояснення, у яку вкладку вона потрапляє і чому;
- які наявні компоненти повторно використані;
- які нові CSS classes додані;
- чи не з'явилися нові inline-style або емодзі.
